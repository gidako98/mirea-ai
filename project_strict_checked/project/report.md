## 2. Постановка задачи и контекст

### 2.1. Предметная область

В службе поддержки входящие обращения отличаются по срочности. Часть обращений — это простые вопросы, часть требует внимания специалистов, а часть может означать инцидент, влияющий на клиентов или безопасность. Автоматическая предварительная классификация приоритета помогает быстрее направить критичные обращения команде реагирования.

Потенциальный пользователь решения — оператор или внутренняя система helpdesk, которая получает новое обращение и хочет быстро оценить его приоритет.

### 2.2. Формулировка задачи в терминах ML

Тип задачи: многоклассовая классификация.

Входные данные:

- `text` — текст обращения;
- `channel` — канал обращения;
- `customer_tier` — тариф клиента;
- `product_area` — зона продукта;
- `sentiment_score` — числовая оценка тональности;
- `account_age_days` — возраст аккаунта клиента.

Целевая переменная:

- `priority`: `low`, `medium`, `high`.

Ограничения:

- модель должна быть достаточно быстрой для HTTP-инференса;
- пайплайн должен быть воспроизводимым;
- результат должен быть понятен на защите, поэтому выбрана интерпретируемая классическая ML-модель, а не тяжёлая нейросеть;
- данные не должны содержать персональных или конфиденциальных сведений.

### 2.3. Метрики качества

Используются две метрики:

- **Accuracy** — общая доля правильных ответов;
- **Macro F1** — среднее F1 по классам без взвешивания по размеру класса.

Macro F1 выбран как ключевая метрика, потому что классы распределены не идеально равномерно: `high` встречается реже, чем `medium` и `low`. Accuracy в такой ситуации может скрывать плохую работу модели на редком, но важном классе.

## 3. Данные

### 3.1. Источник данных

Проект использует синтетические данные, которые генерируются в `src/data.py`. Такой выбор сделан для учебного проекта: он исключает риск попадания в репозиторий персональных данных, коммерческой информации и реальных обращений клиентов.

Файл данных:

- `data/sample_tickets.csv` — демонстрационная выборка из 600 строк.

Данные можно пересоздать командой:

```bash
python -m src.data --output data/sample_tickets.csv --rows 600 --seed 42
```

### 3.2. Структура данных

Ключевые признаки:

| Колонка | Тип | Описание |
|---|---|---|
| `ticket_id` | строка | синтетический идентификатор обращения |
| `text` | текст | краткое содержание обращения |
| `channel` | категория | канал: email, chat, phone, web |
| `customer_tier` | категория | тариф клиента |
| `product_area` | категория | зона продукта |
| `sentiment_score` | число | оценка тональности от -1 до 1 |
| `account_age_days` | число | возраст аккаунта в днях |
| `priority` | категория | целевой класс |

Распределение целевой переменной в текущем датасете:

| Класс | Количество |
|---|---:|
| medium | 247 |
| low | 228 |
| high | 125 |

В генератор добавлен небольшой шум разметки. Это делает задачу менее искусственной: в реальных helpdesk-данных похожие обращения иногда получают разные метки из-за человеческого фактора.

### 3.3. Предобработка и EDA

EDA вынесен в `notebooks/01_eda.ipynb`. В нём проверяются:

- наличие нужных колонок;
- распределение классов;
- базовые статистики по числовым признакам;
- связь признаков с классом приоритета.

Основные выводы:

- классы умеренно несбалансированы;
- текст обращения содержит сильный сигнал для классификации;
- категориальные признаки и числовые признаки добавляют полезный контекст;
- для оценки нужна macro F1, чтобы учитывать качество на каждом классе.

## 4. Модели и подходы

### 4.1. Baseline

В качестве baseline используется `DummyClassifier(strategy="most_frequent")`. Он всегда предсказывает самый частый класс. Такой baseline нужен, чтобы проверить, действительно ли финальная модель учится закономерностям, а не просто повторяет распределение классов.

Результаты baseline на отложенной выборке:

| Модель | Accuracy | Macro F1 |
|---|---:|---:|
| Most frequent class | 0.4133 | 0.1950 |

Baseline показывает низкую macro F1, потому что фактически игнорирует редкие классы.

### 4.2. Финальная модель

Финальная модель реализована как единый scikit-learn `Pipeline`:

1. `TfidfVectorizer` для поля `text`;
2. `OneHotEncoder` для категориальных признаков;
3. `StandardScaler` для числовых признаков;
4. `LogisticRegression` с `class_weight="balanced"`.

Причины выбора:

- модель быстро обучается и быстро работает на инференсе;
- пайплайн сохраняется одним артефактом `model.joblib`;
- текстовые, категориальные и числовые признаки обрабатываются в одном воспроизводимом процессе;
- Logistic Regression проще объяснить на защите, чем более сложные модели;
- `class_weight="balanced"` помогает учитывать различие размеров классов.

### 4.3. Сравнение экспериментов

Результаты финальной модели:

| Модель | Accuracy | Macro F1 |
|---|---:|---:|
| Baseline: most frequent class | 0.4133 | 0.1950 |
| Final: TF-IDF + OneHot + Logistic Regression | 0.9400 | 0.9419 |

Финальная модель выбрана, потому что она значительно превосходит baseline по macro F1 и accuracy, оставаясь простой и воспроизводимой.

Ноутбук с экспериментами: `notebooks/02_experiments.ipynb`. Основные численные результаты также сохранены в `artifacts/metrics.json`.

## 5. Реализация и структура кода

Основной код не находится в одном ноутбуке, а вынесен в модульную структуру `src/`:

- `src/data.py` — генерация, сохранение и загрузка данных;
- `src/modeling.py` — создание baseline, финального пайплайна, оценка и предсказания;
- `src/train.py` — обучение моделей и сохранение артефактов;
- `src/evaluate.py` — оценка сохранённой модели;
- `src/predict.py` — CLI-инференс;
- `src/service.py` — HTTP-сервис;
- `src/config.py` — загрузка конфигурации.

Связь компонентов:

```text
data/sample_tickets.csv
        ↓
src.data.load_dataset
        ↓
src.modeling.build_final_model
        ↓
artifacts/model.joblib + artifacts/metrics.json
        ↓
src.predict / src.service
        ↓
предсказание priority + вероятности классов
```

## 6. Сервис и API

### 6.1. Архитектура сервиса

Сервис реализован в `src/service.py` на стандартной библиотеке Python (`http.server`). Flask/FastAPI не используются, потому что требования проекта говорят о сервисе и endpoint'ах, но не требуют конкретного веб-фреймворка. Такой вариант уменьшает число зависимостей и сохраняет строгое соответствие чеклисту.

Перед запуском сервиса модель загружается из `artifacts/model.joblib`. Endpoint `/predict` использует эту реальную обученную модель, а не заглушку.

### 6.2. Endpoints

#### `GET /health`

Возвращает состояние сервиса:

```json
{
  "status": "ok",
  "model_loaded": true,
  "model_path": "artifacts/model.joblib",
  "service": "ticket-priority-triage"
}
```

#### `POST /predict`

Принимает один объект, список объектов или формат `{"items": [...]}`.

Пример:

```bash
curl -X POST http://127.0.0.1:8000/predict \
  -H "Content-Type: application/json" \
  -d @examples/sample_input.json
```

Ответ содержит предсказанный класс и вероятности классов.

## 7. Наблюдаемость, конфигурация и безопасность

### 7.1. Логи и наблюдаемость

В проекте используется стандартный модуль `logging`. Логируются:

- запуск сервиса;
- путь к модели;
- HTTP-доступ;
- количество объектов в запросе к `/predict`;
- ошибки валидации JSON;
- ошибки предсказания.

Endpoint `/health` позволяет проверить, что сервис работает и модель загружена.

### 7.2. Конфигурация

Основной конфиг: `configs/config.json`.

В него вынесены:

- пути к данным, модели и метрикам;
- random seed;
- размер test split;
- параметры модели;
- host/port сервиса.

Также есть `configs/.env.example` с переменными окружения:

- `AIE_MODEL_PATH`;
- `AIE_SERVICE_HOST`;
- `AIE_SERVICE_PORT`;
- `AIE_LOG_LEVEL`.

### 7.3. Безопасность

Проект не содержит реальных секретов и персональных данных. Файл `.env` добавлен в `.gitignore`, а `configs/.env.example` содержит только безопасные шаблонные значения. Данные синтетические и не позволяют восстановить сведения о реальных людях или компаниях.

Дополнительные заметки описаны в `SECURITY.md`.

## 8. Ограничения и дальнейшая работа

Ограничения:

- датасет синтетический;
- модель не проверялась на реальных обращениях;
- нет полноценной схемы валидации, например через Pydantic;
- нет мониторинга качества после развёртывания;
- нет авторизации, rate limiting и хранения истории запросов.

Дальнейшая работа:

- подключить открытый или разрешённый реальный датасет;
- добавить cross-validation и подбор гиперпараметров;
- добавить мониторинг data drift;
- добавить версионирование моделей;
- расширить API документацией и строгими схемами входа/выхода;
- провести ручной анализ ошибок по классам.

## 9. Сценарий демонстрации на защите

1. Кратко показать структуру проекта и объяснить, где находятся данные, код, отчёт, тесты и артефакты.
2. Открыть `notebooks/01_eda.ipynb` и показать распределение классов.

   ```bash

   ```

6. Запустить сервис:

   ```bash
   python -m src.service --config configs/config.json
   ```

7. Проверить `/health` и `/predict`:

   ```bash
   curl http://127.0.0.1:8000/health
   curl -X POST http://127.0.0.1:8000/predict -H "Content-Type: application/json" -d @examples/sample_input.json
   ```

На защите стоит подчеркнуть, что `/predict` использует сохранённую реальную модель, а не захардкоженную логику.
