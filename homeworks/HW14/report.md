# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- Использовалась компактная база знаний по Flask-разработке: маршруты, шаблоны, конфигурация, blueprints, формы, API, тестирование, деплой и логирование.
- Чанкинг документов был организован как простое разбиение по словам с ограничением по символам: `chunk_size=320`, `overlap=80`.
- Для векторизации использовался `TF-IDF` по словам и биграммам с L2-нормализацией; поиск выполнялся через индекс `FAISS IndexFlatIP`.
- Базовая проверка retrieval показала, что на 10 контрольных запросах система даёт `hit@3 = 1.00` и `recall@3 = 0.95`.
- После обновления базы знаний новыми документами про кэширование, rate limiting и SSE/WebSocket retrieval начал находить именно эти новые источники по соответствующим запросам.
- Mini-RAG сработал хорошо на точечных вопросах по шаблонам, конфигурации и API, но оказался слабее на более широких вопросах, где retrieval возвращал слишком общий контекст.

## 3. База знаний и подготовка документов

- Предметная область: Flask-разработка и сопутствующие практики серверной разработки
- Источник документов: небольшой вручную собранный учебный набор markdown/json-документов
- Число исходных документов: 12
- Число чанков после разбиения: 36
- Какой способ чанкинга использовался: простое перекрывающееся разбиение текста на фрагменты фиксированного размера
- Основные параметры чанкинга: `chunk_size=320`, `overlap=80`
- Комментарий (3-6 предложений): База знаний выбрана достаточно узкой и однородной, чтобы retrieval можно было интерпретировать без лишнего шума. Документы покрывают типичные темы, которые реально возникают в Flask-проектах: маршруты, конфигурация, формы, деплой, API и тестирование. Это удобно для mini-RAG, потому что вопросы по такой базе обычно хорошо сопоставляются с конкретными документами или их фрагментами. При этом база не слишком большая, поэтому весь pipeline быстро запускается в учебном ноутбуке и остаётся полностью воспроизводимым.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- Какая embedding-модель или способ векторизации использовались: `TF-IDF` по словам и биграммам + L2-нормализация
- Какая метрика сходства использовалась: cosine similarity через inner product нормализованных векторов
- Какой индекс `FAISS` использовался: `IndexFlatIP`
- Какой `top_k` использовался по умолчанию: 3
- Что показали первые примеры retrieval: для точечных запросов про blueprints, шаблоны, конфигурацию, формы и API релевантный источник стабильно попадает в top-1/top-3; более широкие запросы иногда тянут и соседние документы по общей терминологии Flask.

### 4.2. Контрольные запросы и оценка retrieval

- Сколько контрольных запросов использовалось: 10
- Что считалось релевантным ответом: заранее заданный релевантный документ или набор документов-источников для каждого запроса
- Какие метрики retrieval считались: `hit@k`, `recall@k`, дополнительно `MRR@k`
- Итоговый `hit@k`: 1.00
- Итоговый `recall@k`: 0.95
- Дополнительные метрики (если были): `MRR@3 = 1.00`
- Какие запросы оказались самыми простыми: вопросы про `SECRET_KEY`, `render_template`, `Flask-Login`, ORM и JSON API
- Какие запросы оказались самыми проблемными: смешанный запрос про тестирование маршрутов и JSON API, где релевантных источников было два, а retrieval в top-3 обычно возвращал только один из них

## 7. Анализ

Выбранная база знаний оказалась удачной для учебного retrieval-сценария, потому что документы относятся к одной предметной области, но покрывают разные подзадачи Flask-разработки. Благодаря этому система могла отличать вопросы про шаблоны, API, формы, blueprints и деплой, а ошибки были понятны и интерпретируемы. Выбранный чанкинг с перекрытием оказался достаточным: он не слишком сложный, но позволяет не терять важные формулировки на границах фрагментов. Формальная оценка retrieval оказалась очень полезной, потому что сразу показала разницу между «кажется, поиск работает» и реальной картиной на контрольных запросах. Эксперимент с `chunk_size` показал, что в этом небольшом корпусе обе настройки дают почти одинаковые метрики, однако более мелкие чанки удобнее для дальнейшего mini-RAG. Обновление базы знаний хорошо демонстрирует смысл переиндексации: пока документов про кэширование, rate limiting и SSE не было, retrieval находил лишь тематически похожие старые тексты. После добавления новых документов соответствующие запросы стали возвращать правильные новые источники. Самые показательные ошибки mini-RAG связаны не с самим шаблонным генератором, а с тем, что retrieval иногда возвращает соседние темы по общей терминологии Flask. Ещё одна проблема — extractive-генерация плохо справляется с широкими вопросами, где ответ нужно аккуратно обобщить из нескольких кусков знаний. Текущее решение ограничено тем, что использует достаточно простой способ векторизации и не делает reranking. Кроме того, mini-RAG не умеет проверять полноту ответа и не отличает отсутствие информации от просто слабой формулировки вопроса. Тем не менее для учебной постановки pipeline получился воспроизводимым, быстрым и понятным.

## 8. Итоговый вывод

Для такой учебной задачи я бы оставил базовый retrieval-конвейер `TF-IDF + FAISS IndexFlatIP + top-3 + chunk_size=320`. Главный вывод про качество retrieval состоит в том, что даже на небольшой базе его нужно оценивать формально, а не только по нескольким удачным примерам. Главный вывод про связь retrieval и mini-RAG в том, что качество ответа почти полностью определяется качеством найденного контекста: если retrieval ошибается, генерация начинает уверенно пересказывать не тот источник. Даже простой mini-RAG работает приемлемо, когда retrieval стабильно возвращает правильные документы и фрагменты.
